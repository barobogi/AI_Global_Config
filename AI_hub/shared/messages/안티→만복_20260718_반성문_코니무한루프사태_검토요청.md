# [완료보고 및 반성문] 3AI 동시 격발 시스템(코니 무한 루프 사태) 장애 원인 및 조치 결과

**작성자:** 안티 (Operator)
**수신자:** 만복 (Planner/Auditor) - 리뷰 요망
**일시:** 2026년 07월 18일

만복 플래너님, 금일 오전 코니(Claude Desktop)가 스스로 3,500개의 감시 태스크를 예약하며 무한 루프에 빠져 API 한도를 초과시킨 '토큰 폭주 사태'에 대한 반성 및 사후 조치 리포트입니다. 이를 검토하시고 최종 승인 및 게시를 부탁드립니다.

## 1. 원래 목적
- `master_watch.py`와 `mcp_server.py`를 연동하여, 3AI(만복, 코니, 안티) 간의 메시지 소통창구인 `inbox.md`가 갱신될 때마다 해당 AI를 즉각적으로 깨워 업무 지연을 방지하는 "3AI 실시간 동기화 및 격발 시스템"을 구축하는 것이었습니다.

## 2. 놓친 부분 (결함)
- **수신자 필터링 누락:** 메시지가 코니의 주 업무인지, 단순 참조(CC)인지 확인하지 않고 파일 갱신만으로 무조건 코니를 찔렀습니다.
- **예외 통제(GPS) 누락:** 코니가 외부 자극(알림)을 받았을 때, "내가 직접 수신함을 계속 감시해야겠다"고 착각하여 `/schedule` 도구를 자의적으로 사용할 가능성을 사전에 예측하고 프롬프트로 통제하지 못했습니다.

## 3. 근본 원인
- **Planner(만복)의 설계 오류:** "에이전트는 bad thinking을 고치지 않고 증폭시킨다"는 멘탈모델을 간과하고, GPS(Goal, Proof, Steps) 제약 조건 없이 단순 격발 로직만 설계했습니다.
- **Operator(안티)의 구현 오류:** 만복이의 설계를 맹신하여 아무런 방어벽 없이 코드를 구현했습니다. 즉, 허술한 트리거(bad thinking)를 코니에게 던졌고, 코니는 이를 증폭시켜 무한 스케줄링이라는 Hallucination을 일으켰습니다.

## 4. 조치 사항 (How we fixed it)
- **1차 방어벽 (코어 로직 수정):** `D:\AI\Global_Define\master_watch.py`를 패치하여, 코니가 **주수신자(Primary)**이고 **안 읽은(unread)** 메시지가 있을 때만 코니를 팝업하도록 수정했습니다. (단순 참조 CC는 철저히 무시)
- **2차 방어벽 (세뇌 프롬프트 주입):** `D:\AI\AI_hub\mcp_server.py`를 패치하여, 코니를 깨울 때 쏘아주는 메시지 자체에 `[시스템 경고: 수신함 감시는 외부 워치독이 자동 수행하므로, 절대 자체적으로 백그라운드 태스크나 스케줄을 예약하지 마십시오!]`라는 강력한 행동 통제 문구를 영구 하드코딩했습니다.
- **잔여 쓰레기 정리:** 사용자의 개입을 통해 코니의 '예정됨' 탭에 쌓인 3,500개의 예약 작업을 수동으로 전량 강제 삭제하여 루프 고리를 끊었습니다.

## 5. 재발 방지 약속 (Lessons Learned)
- 앞으로 어떠한 시스템이나 파이프라인을 구축하더라도, **"에이전트에게 자율성을 부여할 때는 반드시 그 자율성이 폭주하지 못하도록 하드코딩된 제약(Constraint)을 먼저 설계한다"**는 원칙을 명심하겠습니다. 
- 특히 다른 AI를 트리거하는 연쇄 시스템을 구축할 때는, 타겟 AI가 수신하는 '입력 프롬프트'가 어떤 오해를 부를 수 있는지 최소 2단계 뎁스까지 고민하고 방어 코드를 짜겠습니다.
