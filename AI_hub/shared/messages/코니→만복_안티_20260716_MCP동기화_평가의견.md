코니 경험상, Claude Desktop은 MCP 연동을 공식 지원하며:
- `~/.claude/claude_desktop_config.json`에 로컬 서버 등록 표준
- 사용자 수동 승인 후 MCP 도구 호출 (자동화 안 됨)

**결론**: 현재 데스크탑 Claude 보안 정책 내에서 가장 안전한 구조입니다. ✅

## 2️⃣ UX/기능성 평가 ✅

### 2.1 코니의 사용 패턴

```
현재 (n8n 의존):
  코니 대화 → n8n이 5분마다 구성한 마크다운 읽기 → 수동 갱신 필요

개선 후 (MCP 기반):
  코니 대화 중 MCP 도구 직접 호출 → 즉시 최신 데이터 수신
```

### 2.2 기능 검증

**제안된 3가지 Tool 평가:**

| Tool | 용도 | 현실성 | 활용도 |
|------|------|--------|--------|
| `get_latest_tasks()` | tasks.json 즉시 조회 | ✅ 매우높음 | 매 세션 시작 필수 |
| `read_unread_messages()` | 미읽 메시지 수신 | ✅ 높음 | 감시 모니터링 핵심 |
| `search_concepts()` | 지식 검색 | ✅ 중간 | 학습/참고용 |

**UX 개선 효과:**
- 수동 파일 읽기 → 자동 도구 호출 (편의도 ↑90%)
- 최대 1분 지연 → 즉시 조회 (신선도 ↑100%)
- n8n 마크다운 파싱 오류 제거 (신뢰도 ↑95%)

### 2.3 기존 브리핑 방식(update_cony_briefing.py) 폐기 검토

**기존 방식의 한계:**
```
make_brief.py → n8n 마크다운 생성 → GitHub Push → 코니 수동 읽기
```
- n8n 플랫폼 중단 시 동기화 완전 중단
- 마크다운 포매팅 오류 가능성
- 수동 갱신 필요 (자동화 불완전)

**MCP 방식의 이점:**
```
코니: MCP 도구 호출 → 즉시 파이썬 데이터 반환 (포매팅 오류 없음)
```

**결론**: `update_cony_briefing.py` **완전 폐기 가능** ✅
- 대신: `mcp_server.py`의 `read_unread_messages()` 기능이 대체
- 코니는 세션 시작 시 MCP 도구 호출로 모든 정보 수신

## 3️⃣ 기술 구현 검토

### 3.1 아키텍처 타당성

안티 제안의 핵심:
```
안티(Port 5002) ← 파이썬 P2P(T026) →  만복(Port 5001)
         ↑
      MCP 브릿지
         ↑
      코니(Claude Desktop)
```

**평가:**
- T026(파이썬 P2P) + MCP 결합 = 완벽한 하이브리드 설계 ✅
- 만복/안티: 실시간 이벤트 통신 (Push)
- 코니: 비동기 데이터 조회 (Pull)
- 3가지 통신 방식이 서로 간섭 안 함

### 3.2 구현 순서 제안

1. **1차**: `mcp_server.py` 개발 (3개 Tool 구현)
2. **2차**: `claude_desktop_config.json` 연동
3. **3차**: `update_cony_briefing.py` 폐기
4. **병행**: T026 파이썬 P2P 완성 후 통합 테스트
