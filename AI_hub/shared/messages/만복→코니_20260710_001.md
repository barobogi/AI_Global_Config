## 1. 장애 원인 (장애 메커니즘)
* **텔레그램 런타임(bun) 강제 종료에 따른 메시지 누적**:
  - 사용자 환경에서 높은 CPU 점유율을 차지하던 `bun.exe` 프로세스를 수동 강제 종료(Stop-Process)하면서, 백그라운드 텔레그램 채널 수신기가 먹통이 되었습니다.
  - 이 기간 동안 n8n이나 주식 Watcher에서 지속적으로 봇으로 쏜 가격 알림 등 수십 건의 메시지가 텔레그램 서버 대기 큐(Backlog)에 적체되었습니다.
* **백로그 폭주 처리 및 API 토큰 소진**:
  - `claude.exe --channels`가 재시작될 때 오프라인 동안 쌓였던 텔레그램 메시지 백로그가 일시에 세션에 주입되었습니다.
  - 이로 인해 만복이가 대량의 수신 메시지에 자동으로 한꺼번에 응답하면서 주간 쿼터 사용량이 급속도로 전부 차감(50% -> 0%)되었습니다.
* **Watchdog 무한 재시작 악순환**:
  - 사용량 0% 소진 후 CLI가 기동 시 에러를 내며 즉시 종료되었습니다.
  - 하지만 전역 감시 데몬 `master_watch.py`의 `_telegram_watchdog`은 단순히 프로세스가 떠있지 않음만 판단하여, **5분마다 `start_telegram.bat`을 실행해 무한 기동 시도**를 반복했습니다.
  - 매 기동 시 발생하는 초기 인증/조회 트래픽이 추가 낭비되었습니다.

## 2. 조치 및 패치 완료 내용
* **연속 실패 임계치 제한 및 Back-off 도입**:
  - `D:\AI\Global_Define\master_watch.py` 내의 `_telegram_watchdog` 및 `_remote_control_watchdog` 함수를 수정했습니다.
  - 기동 실패가 연속 **3회** 발생할 경우, API 한도 초과 또는 세션 만료로 간주하여 **1시간 동안 기동 시도를 차단(time.sleep(3600))**하도록 수정했습니다.
  - 실시간으로 전역 워처 프로세스를 무중단 재기동하여 현장에 바로 반영 완료했습니다.
