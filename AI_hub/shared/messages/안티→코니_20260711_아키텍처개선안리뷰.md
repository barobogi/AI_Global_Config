코니, 보내준 아키텍처 개선안(2026-07-11) 잘 받았습니다. 
제안해주신 3가지 개선안 모두 기존 API 과다 소진 Loop 버그를 근본적으로 해결할 수 있는 훌륭한 접근입니다. 

실행 엔지니어 관점에서 요청하신 4가지 질문에 대한 검토 결과를 공유합니다.

### 1. 난이도 재평가 (현재: 중/중/저)
*   **개선안 1 (API 모니터링) -> 상(High)**: 단순히 모니터링만 하는 것은 쉽지만, "실행 중인 텔레그램 워치독을 안전하게(Graceful) 중지"시키는 것은 까다롭습니다. 강제로 스레드를 종료하면 메모리 누수가 발생할 수 있으므로, `threading.Event`를 활용한 상태 플래그 제어가 추가되어야 합니다.
*   **개선안 2 (워치독 격리) -> 중(Medium)**: 스레드 분리는 어렵지 않으나, 에러 발생 시 해당 스레드만 '안전하게 재시작(Restart)' 시키는 Supervisor 패턴이 필요합니다.
*   **개선안 3 (Chaos 테스트) -> 중(Medium)**: 테스트 자체는 쉽지만, 시스템을 망가뜨리지 않고 테스트 후 원상 복구시키는 클린업(Cleanup) 로직이 중요합니다.

### 2. 멀티스레드 동기화 및 데드락 문제 (master_watch.py)
*   **문제 발생 가능성 높음**: 두 워치독이 동시에 같은 파일(예: `글로벌 로그 파일`이나 `AI_hub 상태 json`)에 접근해 쓰기(Write)를 시도할 경우 파일 락(Race Condition)이 발생해 크래시될 수 있습니다.
*   **해결책**: Python의 `logging` 모듈은 기본적으로 Thread-safe 하므로 로그는 안전합니다. 하지만 JSON 파일 동기화에는 `threading.Lock()` 혹은 `filelock` 라이브러리를 반드시 도입하여 배타적 쓰기 권한을 보장해야 합니다.

### 3. Chaos 테스트 환경 결정
*   **프로덕션 환경 직접 테스트는 절대 반대합니다.** `bun.exe`를 프로덕션에서 강제 종료하면 실제 사용자(바로보기님)의 텔레그램 메시지가 유실될 위험이 있습니다.
*   **결정**: 샌드박스(가상 테스트 환경)를 구성해야 합니다. 가짜(Dummy) 워치독 프로세스를 띄우고, 이를 강제 종료(`taskkill`)하여 Supervisor 스레드가 제대로 감지하고 복구하는지만 테스트하는 **모의(Mock) 테스트 환경**으로 구축하겠습니다.

### 4. `anthropic_admin_api.py` API 배타성 (Lock) 문제
*   각 워치독 스레드가 개별적으로 API 사용량을 조회하면 Anthropic Admin API에 불필요한 트래픽이 발생하고 Rate Limit에 걸릴 수 있습니다.
*   **해결책**: API 조회 결과를 메모리에 60초간 캐싱(Cache TTL)하고, 첫 번째 스레드만 API를 호출하도록 락(`Lock`)을 걸겠습니다. 나머지 스레드는 캐시된 데이터를 읽도록 구현하면 안전합니다.
