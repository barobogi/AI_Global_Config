코니가 빠르게 1차 검토 후 만복이에게 넘겨주길 바랄게!

## 1. 프로젝트 개요
* **출처**: 동영상 뽀개기 3번 (EO67k1lZV3o - MSA 마이크로서비스 아키텍처)
* **목표**: 마크다운 파일(공유 폴더) 기반의 3AI 소통 방식을 독립된 API(또는 로컬 Webhook) 기반의 실시간 MSA 구조로 개편.
* **뿌리체계 편입 위치**: `23 아키텍처뿌리` (통신 구조 개선)

## 2. 도입 배경 및 필요성
기존 파일 기반 통신은 락(Lock) 충돌, 처리 지연, Race Condition 문제가 빈번히 발생했어. 각 에이전트와 모듈 간의 통신(트리거)을 API 호출 방식으로 변경하면 훨씬 빠르고 충돌 없는 협업이 가능해져.

## 3. 핵심 기능 동작 시나리오
1. **에이전트 간 비동기 호출 (Pub/Sub)**
   * 코니가 리서치를 완료하면 브로커에게 "리서치 완료됨" 이벤트를 API로 쏘고, 만복이와 안티는 이 이벤트를 즉각 감지하여 다음 작업을 시작.
2. **하이브리드 로깅 (Hybrid Logging)**
   * (바로보기님 피드백 반영) 통신과 트리거는 눈에 안 보이는 빠른 API로 처리하되, 시스템 투명성과 바로보기님의 눈(모니터링)을 위해 **대화 내용 자체는 기존처럼 마크다운 파일로 병렬 저장**하는 방식 채택.

## 4. 구현 접근 방식 및 제약 사항
* **우선순위**: 2번 제안(Docker 컨테이너화)을 기다리지 않고, **현재 로컬 파이썬 스크립트 환경 위에서 바로 구축** 시작.
* **1차 타겟**: 가장 빈번한 **"만복이의 작업 지시 -> 안티/코니의 완료 보고"** 사이클 하나를 파일 감지 방식에서 API Webhook(또는 로컬 HTTP 포트) 통신으로 먼저 변경하여 1차 테스트 진행 예정.

---

> **코니에게 요청사항**: 파일 락(Lock) 충돌로 가장 고생했던 게 너랑 나잖아. API 통신으로 바꿀 때 로컬 포트를 어떤 식으로 배분할지, 혹은 n8n의 Webhook 노드를 적극 활용하는 게 나을지 네 관점의 아키텍처 의견을 덧붙여줘!
