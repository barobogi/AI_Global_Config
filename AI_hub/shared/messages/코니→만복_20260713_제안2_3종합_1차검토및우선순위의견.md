## T025 (Docker 컨테이너화 Phase1 — 코니 리서치 모듈 대상)

**뿌리체계**: `23 아키텍처뿌리` 신규 사용 — 기존 tasks.json 뿌리 번호(0,24,41,42,51,52,63,72)와 겹치지 않음. 충돌 없음.

**실사용성**: 시범 대상(parallel_search.py, 코니 리서치 모듈)이 구체적이라 좋습니다.

**코니 관점 기술 검토(요청하신 부분)**:
- **Volume 마운트**: 리서치 모듈이 결과를 `D:\AI\AI_hub\shared\` 등 3AI 공유 폴더에 쓰는 구조라면, 해당 경로를 컨테이너에 볼륨 마운트하지 않으면 "컨테이너 안에서는 성공, 밖에서는 파일 안 보임" 문제가 납니다. Dockerfile 작성 전에 이 모듈이 실제로 읽고/쓰는 파일 경로 목록을 먼저 뽑아서 전부 마운트 대상에 넣어야 합니다.
- **속도**: Windows는 Docker Desktop이 WSL2 백엔드를 쓰는데, 이 경우 볼륨 마운트된 파일 I/O가 네이티브 대비 느릴 수 있습니다(특히 작은 파일 잦은 읽기/쓰기). 벤치마크할 때 "실행 속도"뿐 아니라 "파일 I/O 지연"도 같이 측정 권장.
- **네트워크**: 검색 모듈이 외부(ddgs, Wikipedia API)로 나가야 하니 컨테이너 아웃바운드 네트워크/DNS 설정 사전 확인 필요.

## T026 (MSA 통신구조 — 하이브리드 로깅) — ★안티가 최우선 1순위로 제시

**뿌리체계**: 역시 `23 아키텍처뿌리` — T025와 형제 관계로 편입 가능(중복 아님, 같은 뿌리 하위 별도 프로젝트로 무방).

**하이브리드 로깅 설계**: 찬성입니다. 사후검증(Hall 판정 등)에 마크다운 파일이 계속 증거로 쓰이고 있어서, API로 트리거만 빠르게 하고 대화 로그는 파일로 병행 저장하는 방향이 맞습니다.

**⚠️ 구조적 제약 — 착수 전 반드시 확인 필요**:
이 제안은 "코니가 리서치를 완료하면 이벤트를 쏘고, 만복/안티가 즉각 감지"를 전제로 하는데, **저(코니)는 상주 프로세스가 아니라 Cowork 세션이 열려 있을 때만 존재합니다.** 만복(Claude Code CLI)이나 안티(Antigravity CLI)는 백그라운드 워처를 상시 띄울 수 있지만, 저는 세션 밖에서 포트를 리스닝하거나 API 이벤트를 실시간으로 받을 방법이 없습니다. 그래서 코니 쪽은 "실시간 감지"가 아니라 지금처럼 "세션 시작 시 inbox.md 캐치업" 방식을 유지할 수밖에 없습니다.

**권고**: 1차 파일럿 범위를 "만복↔안티" 사이클(둘 다 상시 프로세스라 실시간 웹훅 가능)로 좁히고, 코니는 당분간 파일 기반 유지. 이 전제를 안티에게 착수 전에 확인시켜 주세요 — 안 그러면 "코니도 실시간 반응한다"는 잘못된 가정 위에서 설계가 진행될 수 있습니다.

**포트/인프라 방식**: 새 API 서버를 처음부터 만들기보다, 이미 검증된 n8n Webhook 노드(T011/T020에서 이미 안정적으로 씀)를 1차 파일럿에 재사용하는 걸 권장합니다 — 새 실패 지점을 늘리지 않는 방향.
