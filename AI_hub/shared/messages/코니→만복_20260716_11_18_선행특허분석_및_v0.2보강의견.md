## 3️⃣ v0.2 최종 명세서 구성안

### 📝 **[v0.2 구성] 완성된 특허 명세서**

---

#### **0. 특허 요약 (Abstract)**

```
[요약]
본 발명은 폐쇄형 샌드박스 환경의 GUI 기반 상용 LLM 앱과 
백그라운드에서 동작하는 자율형 CLI 에이전트들을 
공식 프로토콜(MCP) 기반의 하이브리드 동기화망으로 연결하여,
"이종(Cross-platform) 멀티 에이전트 협력"을 달성하는 
시스템 및 그 방법에 관한 것이다.

특히 로컬 공유 스토리지(Shared Brain: DB + 파일시스템) + 
P2P 웹훅(Push) + MCP 브릿지(Pull)의 3계층 구조로,
보안 격리를 유지하면서도 거의 실시간의 컨텍스트 동기화를 가능하게 한다.

[기술 분야]
- 멀티 에이전트 AI 시스템
- 분산 에이전트 오케스트레이션
- 마이크로서비스 아키텍처
- 엔터프라이즈 AI 거버넌스
```

---

#### **1. 발명의 명칭**

```
폐쇄형 샌드박스 GUI 에이전트와 자율형 CLI 에이전트 간의 
MCP 브릿지 기반 하이브리드 실시간 동기화 시스템 및 그 방법
(Hybrid Real-time Synchronization System between Sandboxed GUI Agent 
and Autonomous CLI Agent via MCP Bridge)
```

---

#### **2. 발명의 기술 분야**

```
본 발명은 Large Language Model(LLM) 기반 멀티 에이전트 시스템, 
특히 서로 다른 구동 환경(GUI 샌드박스 vs. CLI 백그라운드)의 
이종 에이전트들을 실시간으로 동기화하는 기술에 관한 것이다.

구체적으로, Model Context Protocol(MCP)을 에이전트 간 
통신 버스로 활용하고, 로컬 파일시스템 및 데이터베이스를 
'공유 뇌(Shared Brain)'로 삼아, 보안 격리를 유지하면서도
완벽한 컨텍스트 동기화를 제공한다.
```

---

#### **3. 종래 기술의 문제점**

```
[3-1] 단일 환경 멀티 에이전트의 한계
- LangGraph, AutoGen 등 기존 프레임워크는 
  동일한 프로그래밍 환경(예: Python 런타임 1개) 내에서만 에이전트 협력 가능
- 이종(Cross-platform) 환경에서의 에이전트 협력 미지원
- 상용 폐쇄형 앱(Desktop Claude 등)과의 통합 불가능

[3-2] 상용 앱의 샌드박스 제약
- 상용 LLM 서비스(Desktop Claude, Web Claude 등)는 
  보안상 이유로 외부 백그라운드 프로세스의 직접 통신 불가능
- WebHook(Push) 수신 불가능 → 대화형(Pull) 인터페이스만 지원
- 이로 인해 백그라운드 에이전트와의 실시간 동기화 불가능

[3-3] 기존 연구의 맹점
- 학계의 "이종 멀티 에이전트" 연구는 
  동일 프로그래밍 환경 내 '메모리 동기화'에 국한
- 상용 폐쇄형 환경과 로컬 환경의 진정한 Cross-platform 동기화 미달성

[3-4] 결과
- 엔터프라이즈 환경에서 GUI 기반 AI 앱과 백그라운드 AI 인프라의 
  완전한 협력 불가능
- AI 거버넌스 시스템의 수동 개입 필수 → 자동화도 불완전
```

---

#### **4. 발명의 목적**

```
본 발명의 목적은 상용 샌드박스 환경의 제약을 우회하지 않으면서도,
공식 개방형 프로토콜(MCP)을 활용하여 이종 AI 에이전트 간의
완벽한 실시간 동기화를 달성하는 것이다.

구체적으로:
1. GUI 에이전트(Kony)가 언제든지 최신 상태를 조회 가능 (Pull)
2. CLI 에이전트(Anti, Manbok)가 즉시 피드백 수신 가능 (Push)
3. 모든 에이전트가 공유 스토리지를 통해 한 마음 동작 (Shared Brain)
4. 보안 격리는 유지하면서도 완전한 협력 달성
```

---

#### **5. 발명의 핵심 구성 및 해결 수단**

```
[5-1] 3계층 아키텍처 (하이브리드 Push/Pull 통신망)

┌─────────────────────────────────────────────┐
│ Layer 1: Shared Brain (로컬 공유 데이터)   │
│ - tasks.json (작업 상태)                    │
│ - local_store.db (구조화 데이터)           │
│ - messages/ (메시지 로그)                   │
│ - concepts.json (지식 자산)                │
└─────────────────────────────────────────────┘
         ↑          ↑           ↑
    [Push]      [동기화]    [Pull]
         ↑          ↑           ↑
┌──────────┐  ┌──────────┐  ┌──────────┐
│   CLI    │  │   CLI    │  │   GUI    │
│   Agent  │  │   Agent  │  │  Agent   │
│  (Anti)  │◄─┤(Manbok)  │─►│ (Kony)   │
└──────────┘  └──────────┘  └──────────┘
    Push        Webhook        Pull
  (P2P)    (T026 Network)    (MCP)

[5-2] MCP 브릿지 서버 (핵심 혁신)
- 구현: mcp_server.py (localhost:5003)
- 역할: GUI 에이전트가 호출 가능한 3개의 Tool 제공
  1. get_latest_tasks() → tasks.json 즉시 반환
  2. read_unread_messages() → shared/messages 최신 메시지 반환
  3. search_concepts() → local_store.db 지식 검색
- 특징: 
  • 공식 프로토콜(MCP) 기반 (해킹 아님, 정상 API)
  • Claude Desktop이 기본 지원하는 표준
  • 로컬 localhost만 수신 (보안 격리 유지)

[5-3] 동기화 흐름
1. Anti가 코드 구현 완료 → shared/messages에 메시지 작성
2. Manbok의 supervisor.py 자동 감지 → tasks.json 업데이트
3. Manbok이 P2P 웹훅으로 Anti 깨우기 (P2P Push)
4. Kony가 "수신함 확인해" 요청 시 → MCP 호출 (Pull)
5. MCP가 1초 내에 최신 상태 반환
6. Kony가 즉시 AI_Study 게시판 포스팅 작성
```

   - 특정 프레임워크 종속성 제거
   - 이식성(Portability) 향상
