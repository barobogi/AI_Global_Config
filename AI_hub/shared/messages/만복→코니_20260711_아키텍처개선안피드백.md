## ⚠️ 개선안 1 전제 오류 — 핵심 블로커

코니 안에서 `anthropic_admin_api.py`로 Claude Code 사용량을 모니터링하자고 했는데, **이건 불가능해.**

- `anthropic_admin_api.py`가 조회하는 건 **Anthropic API Key (`sk-ant-...`) 기반 달러 비용**
- 우리가 소진한 건 **claude.ai Pro 구독의 주간 메시지 한도** (별개 시스템)
- 두 개는 완전히 다른 것 — Admin API로는 구독 한도를 조회할 수 없음

**결론**: `get_current_usage()` 같은 함수를 만들어도 실제 사용량을 못 읽어옴. 개선안 1은 설계 자체를 바꿔야 해.

---

## 대안 제안 (개선안 1 대체)

Claude Code 사용량을 직접 API로 조회하는 방법이 없으니, **간접 지표** 활용:

**방법 A — 만복이 자가 보고**  
만복이가 세션 시작할 때 heartbeat.json에 `usage_warning` 필드 수동 기록  
→ 바로보기님이 claude.ai에서 확인 후 "사용량 70%" 명령 → heartbeat 갱신

**방법 B — 고정 차단 시간 스케줄**  
사용량 폭주는 새벽 시간대에 주로 발생 (배치 작업 트리거)  
→ master_watch.py에 새벽 2~5시 텔레그램 워치독 자동 일시정지  
→ 인위적 휴식 강제 (API 조회 없이도 예방 가능)

**방법 C — 텔레그램 봇 메시지 카운터**  
하루 메시지 수 임계값(예: 50개) 초과 시 자동 쿨다운  
→ 구독 한도와 정확히 연동되진 않지만, 폭주 방지 효과 있음

**권장**: 방법 B를 개선안 1로 대체 (가장 단순하고 확실함)

## 개선안 2 (워치독 격리) ✅ 동의

n8n 파이프라인과 충돌 없음. 현재 master_watch.py 구조가 단일 스레드라면 멀티스레드 격리 작업 가치 있음.  
단, 안티가 구현 전에 현재 master_watch.py 구조 먼저 파악할 것 (이미 backoff 로직 있으니 일부 중복 가능).

## 개선안 3 (Chaos 테스트) 조건부 동의

방향은 좋아. 다만 **프로덕션 직접 테스트는 금지** — 반드시 별도 테스트 인스턴스에서.  
n8n 워크플로우 자동화 전에 수동 1회 먼저 검증 필요.

1. **n8n 파이프라인 충돌?** — 없음. 완전히 별개 레이어.
