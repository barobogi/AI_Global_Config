# 아키텍처 개선 2 — 워치독 격리 (Supervisor 패턴) 설계서

작성: 만복 / 2026-07-12

---

## 1. 현재 구조 (문제점)

```
master_watch.py (단일 프로세스)
├── [Thread] auto-discover        — git 저장소 자동 감지
├── [Thread] daily-scheduler      — GeekNews/코니싱크/개념노트 스케줄
├── [Thread] aihub-heartbeat      — 5분마다 heartbeat + 브리핑 갱신
├── [Thread] telegram-watchdog    — 두복이(claude.exe --channels) 감시
└── [Main]   watchdog Observer    — 파일 변경 감지 → git push
```

**문제점 3가지:**
1. master_watch.py 자체가 죽으면 워치독 포함 전체 사망
2. 워치독이 파일 감시 + 프로세스 감시 + 스케줄링을 혼재 담당 — 책임 범위 초과
3. T020 완료 후 `n8n_telegram_bot.py`도 감시 대상 추가 → 더 복잡해짐

---

## 2. 개선 후 구조 (Supervisor 패턴)

```
supervisor.py  ← 부팅 시 1개만 실행 (Windows 작업 스케줄러)
├── [감시] master_watch.py    — 죽으면 자동 재시작 (30초 간격 체크)
└── [감시] n8n_telegram_bot.py  — 죽으면 자동 재시작 (T020 완료 후 추가)
```

**핵심 원칙:**
- supervisor.py는 "프로세스 생사 감시" 만 담당 (파일 감시 X, 스케줄 X)
- master_watch.py는 "파일 감시 + 스케줄" 만 담당 (워치독 스레드 제거)
- 텔레그램 워치독(`_telegram_watchdog`)은 supervisor.py로 이관
- master_watch.py가 죽어도 supervisor.py가 살아있어 재시작 보장
