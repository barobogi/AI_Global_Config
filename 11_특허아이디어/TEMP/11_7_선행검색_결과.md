## 2. 선행기술 공백 — 본 발명만의 신규성 포인트

아래 조합은 기존 선행기술에서 **발견되지 않은 구성**입니다.

| # | 신규성 포인트 | 선행기술 대비 |
|---|-------------|-------------|
| A | **이기종 AI 채널 간 단일 페르소나 공유** (벤더 무관) | Amazon/Google/Samsung 모두 자사 에코시스템 내부. 이기종 결합 없음 |
| B | **채널별 명시적 역할 분리 설계** (알림채널 / 원격지시채널 / 깊은작업채널 / 자동화채널) | 기존 멀티채널 봇은 동일 기능을 여러 채널에 복제. 역할 분리 개념 없음 |
| C | **AI가 상황에 따라 최적 채널 자동 선택 (AI 라우팅)** | 기존은 사용자가 채널 선택. AI가 라우팅 판단하는 구조 없음 |
| D | **파일 시스템 기반 플랫폼 독립적 컨텍스트 인계** (코니_sync.md, REF 구조) | Apple Handoff는 동일 벤더 전용. 범용 파일 기반 인계 메커니즘 없음 |
| E | **물리 기기(인형/스마트홈) 포함 HW+SW 이기종 채널 통합** | 스마트 스피커 계열은 존재하나, 인형+CLI+봇+IoT를 단일 페르소나로 통합하는 구조 없음 |
| F | **개인 사용자 수준의 오케스트레이션 자가 구축 (No-code/Low-code 방식)** | 엔터프라이즈 수준 오케스트레이션(AWS Step Functions 등)은 존재. 개인용 자가 구축 모델 없음 |

### 핵심 신규성 조합 (3가지 동시 충족 시 신규성 성립)
```
이기종 AI 채널 + 단일 페르소나 공유 + AI 자동 라우팅
```
- 세 요소 중 2개씩만 결합한 선행기술은 존재
- **세 요소 동시 결합**은 선행기술 미발견

## 3. 특허 가능성 판단

### 종합 판단: 🔶 중간~중상

| 평가 항목 | 판단 | 근거 |
|-----------|------|------|
| **발명 성립성** | ✅ 통과 | SW 방법 발명 + HW 채널 포함 시 물건 발명 가능 |
| **신규성** | 🔶 조건부 통과 | 개별 구성요소는 공지기술이나 조합은 신규성 주장 가능 |
| **진보성** | 🔶 조건부 통과 | "AI 자동 라우팅" + "이기종 채널 페르소나" 조합이 비자명성 핵심 |
| **산업상 이용가능성** | ✅ 통과 | 실제 구현 가능 (바로보기 시스템이 프로토타입 증명) |

### 가능성이 '중간~중상'인 이유

**긍정 요인:**
1. 이미 동작하는 프로토타입 존재 (바로보기+만복이+두복이+세복이 시스템)
2. 이기종 채널 결합 + AI 자동 라우팅 조합이 선행기술 미발견
3. 물리 기기(호두 AI 인형) 포함 시 HW+SW 결합으로 특허 성립성 강화
4. IoT(스마트홈) 채널 추가 시 산업적 적용 범위 확대

**부정 요인:**
1. 순수 SW 아키텍처로 출원 시 한국/미국 모두 거절 위험 (추상적 아이디어)
2. Amazon/Google의 대규모 유사 특허 포트폴리오가 선행기술 검토 부담
3. "단일 페르소나" 개념이 아직 법적으로 명확히 정의되지 않아 청구항 범위 설정 어려움
4. 멀티채널 봇 프레임워크 관련 공개기술(오픈소스) 다수 존재

## 4. 핵심 청구항 개선 제안

### 현재 청구항 후보 (v0.1 기존)
> "복수의 이기종 AI 에이전트 채널이 단일 사용자 컨텍스트와 AI 페르소나를 공유하며, 채널별 역할 분리 및 자동 라우팅을 통해 사용자 경험의 연속성을 보장하는 퍼스널 AI 오케스트레이션 시스템 및 방법"

**문제점:**
- "이기종", "페르소나 공유", "자동 라우팅" 개념이 추상적
- 기술적 구성요소가 명시되지 않아 거절 위험

---

### 개선된 독립항 (광의) — 시스템 청구항

```
복수의 이기종 AI 채널 인터페이스를 포함하는 퍼스널 AI 오케스트레이션 시스템으로서,

(a) 각 AI 채널 인터페이스에 공통으로 제공되는 단일 사용자 컨텍스트 저장소 — 
    사용자의 진행 중 작업, 대화 이력, 선호 설정을 포함하는 구조화된 데이터 형식으로 저장;

(b) 각 AI 채널 인터페이스에 고유하게 정의된 역할 프로파일 — 
    채널별 처리 가능한 요청 유형, 응답 형식, 우선순위를 포함;

(c) 입력된 요청의 특성(긴급도, 복잡도, 사용자 현재 위치, 기기 가용성)을 분석하여
    상기 역할 프로파일에 기반해 최적 채널을 자동 선택하는 AI 라우팅 모듈;

(d) 제1 채널에서 제2 채널로의 전환 시, 상기 컨텍스트 저장소의 데이터를
    제2 채널의 AI 에이전트에 자동으로 전달하는 세션 핸드오프 모듈;

을 포함하며, 상기 복수의 AI 채널 인터페이스는 서로 다른 벤더의 AI 서비스,
메시징 플랫폼, 자동화 워크플로우 엔진, 및 물리적 IoT 기기 중 둘 이상을 포함하는 것을 특징으로 하는 시스템.
```

---

### 종속항 1 — AI 자동 라우팅 알고리즘 구체화

```
제1항에 있어서, 상기 AI 라우팅 모듈은
요청의 예상 처리 시간이 임계값을 초과하는 경우 고처리 능력 채널로,
사용자의 모바일 기기 활성 상태 감지 시 모바일 채널로,
기 설정된 시간 조건 충족 시 자동화 워크플로우 채널로
우선순위에 따라 순차 선택하는 규칙 기반 라우팅 결정 트리를 포함하는 것을 특징으로 하는 시스템.
```

---

### 종속항 2 — 파일 시스템 기반 컨텍스트 인계 (플랫폼 독립성)

```
제1항에 있어서, 상기 컨텍스트 저장소는 특정 벤더 플랫폼에 종속되지 않는
로컬 파일 시스템 또는 범용 클라우드 스토리지에 구조화된 텍스트 형식으로 저장되며,
파일 시스템 감시 모듈이 컨텍스트 갱신을 감지하여 연결된 모든 AI 채널에
자동으로 동기화 이벤트를 전달하는 것을 특징으로 하는 시스템.
```

---

### 종속항 3 — 물리 기기(HW) 채널 포함 (호두 AI 결합)

```
제1항에 있어서, 상기 복수의 AI 채널 인터페이스는
음성 입출력 모듈과 무선 통신 모듈을 탑재한 물리적 완구 또는 IoT 기기를 포함하며,
상기 물리적 기기의 AI 채널은 상기 컨텍스트 저장소와 실시간 동기화되어
사용자가 소프트웨어 채널에서 물리 채널로 전환 시 동일한 AI 페르소나와
연속적 대화를 유지할 수 있는 것을 특징으로 하는 시스템.
```

---

### 종속항 4 — 채널 부재 시 에스컬레이션 (장애 대응)

```
제1항에 있어서, 상기 AI 라우팅 모듈은
최우선 채널이 응답 불가 상태일 경우 차순위 채널로 자동 에스컬레이션하며,
에스컬레이션 이력 및 원인을 상기 컨텍스트 저장소에 기록하여
장애 복구 후 원래 채널로 복귀 시 이력을 자동 인계하는 것을 특징으로 하는 시스템.
```

---

### 독립항 (협의) — 방법 청구항

```
복수의 이기종 AI 채널을 단일 사용자 페르소나로 운용하는 AI 오케스트레이션 방법으로서,

(a) 사용자 요청을 수신하는 단계;
(b) 요청의 긴급도, 복잡도 및 현재 사용자 환경을 분석하는 단계;
(c) 분석 결과와 각 AI 채널의 역할 프로파일을 비교하여 최적 처리 채널을 선택하는 단계;
(d) 선택된 채널의 AI 에이전트에 현재 컨텍스트를 전달하여 요청을 처리하는 단계;
(e) 처리 결과 및 갱신된 컨텍스트를 공유 저장소에 기록하는 단계;

를 포함하며, 상기 복수의 이기종 AI 채널은 동일한 AI 엔진을 공유하지 않아도 되며
각 채널의 AI 서비스가 서로 다른 제공자(provider)의 것이어도 동일한 페르소나 정의를
상기 공유 저장소를 통해 참조함으로써 일관된 사용자 경험을 제공하는 것을 특징으로 하는 방법.
```
