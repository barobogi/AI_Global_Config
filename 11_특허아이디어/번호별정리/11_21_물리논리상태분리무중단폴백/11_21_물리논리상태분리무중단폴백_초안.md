# 11_21 — 물리적 상태와 논리적 상태를 분리한 무중단 연쇄 폴백

**버전**: v0.1 (2026-07-18)  
**상태**: 초안 등록  
**작성**: 만복 (코니 검토 반영)  
**신규성**: ⭐⭐⭐⭐⭐  

---

## 1. 핵심 아이디어

멀티 에이전트 시스템에서 에이전트의 **물리적 상태**(창/프로세스 실행 여부)와 **논리적 상태**(실제 응답 가능 여부: `is_active`)를 독립적으로 관리하여, 좀비 에이전트 감지 및 무중단 연쇄 폴백(Cascading Fallback)을 실현하는 아키텍처.

---

## 2. 배경 / 기존 기술의 한계

| 항목 | 기존 시스템 | 본 발명 |
|------|------------|---------|
| 상태 판단 | 창 열림 = 살아있음 | 창 열림 + `is_active=true` 필수 |
| 문제 감지 | 타임아웃만 감지 | Session Limit 등 논리적 무응답도 감지 |
| 대응 | 재시도 또는 실패 | 우선순위 체인에 따른 무중단 자동 폴백 |
| 제어 | 프로세스 종료/시작만 가능 | 원격으로 논리 상태만 토글 가능 (DND 모드) |

- Kubernetes Circuit Breaker: 물리 상태(Pod Running)만 감지, 논리적 활성화 개념 없음
- 기존 retry logic: passive 재시도, active 원격 제어 불가

---

## 3. 기술 구성

### 3-1. 에이전트 레지스트리 (agent_registry.json)
```json
{
  "manbok": { "fallback_priority": 1, "is_active": true },
  "kony":   { "fallback_priority": 2, "is_active": true },
  "anti":   { "fallback_priority": 3, "is_active": true },
  "nvidia_nim": { "type": "api", "fallback_priority": 4, "is_active": true }
}
```
- `is_active`: 논리적 활성 상태 (물리 상태와 독립)
- `fallback_priority`: 폴백 우선순위

### 3-2. MCP 브릿지 서버 (mcp_server.py)
- `/trigger` 엔드포인트: `is_active=false` 시 즉시 404 반환 → 폴백 트리거
- 물리적으로 창이 열려 있어도 논리 상태 우선

### 3-3. 연쇄 폴백 엔진 (master_watch.py)
```
1위 만복 → 실패(404) → 2위 코니 → 실패(404) → 3위 안티 → 실패(404) → 4위 NIM(API)
```
- 모든 에이전트 실패 시: 텔레그램 경고 + NIM이 inbox 분석 요약 발송

### 3-4. 원격 상태 제어
- 텔레그램 명령: `/sleep 코니`, `/wake 만복`
- `is_active` 값만 토글 → 물리적 프로세스 종료 없이 논리적 DND 구현

---

## 4. 신규성 포인트

1. **물리↔논리 이중 상태 분리**: 기존 시스템에서 찾기 어려운 개념
2. **좀비 에이전트 감지**: 창이 켜져 있어도 Session Limit 등으로 무응답인 상태를 논리 상태(404)로 감지
3. **우선순위 기반 무중단 체인**: 실패 즉시 다음 우선순위로 자동 전환
4. **이종 에이전트 혼합 폴백**: UI 기반(코니/안티) + API 기반(NIM) 혼합 체인
