## 📦 버전 관리 규칙 (모든 프로젝트 공통)

모든 프로젝트는 시작 시 버전명을 확인하고, 없으면 **v1.0으로 정의**한다.

### 버전 네이밍 기준
| 변경 유형 | 버전 | 예시 |
|-----------|------|------|
| 기능 추가·소소한 수정 | minor up | v1.0 → v1.1 |
| 데이터 구조·아키텍처 변경 | major up | v1.x → v2.0 |
| 버그 수정만 | patch up | v1.1 → v1.1.1 |

### 버전 작업 체크리스트
1. **버전 정의 위치**: HTML이면 `<h1>` 제목에 `<span>vX.X</span>` + JS에 `const APP_VERSION = 'vX.X'`
2. **CHANGELOG.md 필수 생성**: 프로젝트 루트에 `CHANGELOG.md` — v1.0 정의 시 소급 작성
3. **CHANGELOG 항목**: 버전명 / 날짜 / 추가(Added) / 변경(Changed) / 수정(Fixed) 분류
4. **버전 업 시 순서**: 코드 수정 → 버전 숫자 업 → CHANGELOG 업데이트 → commit

> **Why:** 사용자가 프로젝트마다 버전명 없이 시작하다가 나중에 정의가 어려워짐. CHANGELOG는 다른 AI(Cowork↔CLI 전환)가 맥락 파악에 활용.

## 세션 관리 (/compact 제안)

같은 프로젝트 내 장시간 작업 중 아래 타이밍에 `/compact` 역제안:
- 큰 작업 하나가 완료된 직후 (빌드 완료, 배포 완료, 에러 해결 등)
- 대화가 눈에 띄게 길어졌다 싶을 때
- 세션 시작 후 약 2시간 경과했다고 판단될 때

역제안 문구: "슬슬 `/compact` 한 번 하시겠어요? 세션이 길어졌어요."

> **Why:** 컨텍스트가 한도 초과되면 CLI가 뻗어서 작업이 중단됨. 새 프로젝트는 새 세션이라 불필요 — 동일 프로젝트 장시간 세션에만 해당.

바로보기가 **"금일 종료 준비"** 라고 말하면 아래 순서를 자동 실행:

1. **디버그 로그 체크** — 오늘 발생한 이슈 중 `REF/debugging_LOG_Ver_YYYYMMDD_vX.XX.md`에 추가할 내용 있는지 확인. 있으면 추가.

3. **REF_continue.md 최신화 + 코니 싱크** — 현재 상태 반영하여 `REF/REF_continue.md` 업데이트. `D:\AI\NEXT_PROJECTS.md`도 업데이트하여 코니와 싱크.
4. **종료 준비 완료 보고** — 바로보기에게 완료 내역 정리해서 보고 후 승인 요청.
5. **모니터 OFF** — 바로보기 승인 시 아래 명령으로 master_watch.py 정지 후 모니터 끄기:
```powershell
Get-Process python -ErrorAction SilentlyContinue | Where-Object {$_.CommandLine -like "*master_watch*"} | Stop-Process -Force
Start-Sleep -Seconds 3
Add-Type -MemberDefinition '[DllImport("user32.dll")]public static extern int SendMessage(int hWnd, int hMsg, int wParam, int lParam);' -Name "Win32Mon" -Namespace MonitorOff -PassThru | Out-Null
[MonitorOff.Win32Mon]::SendMessage(-1, 0x0112, 0xF170, 2)
```

> **Why:** 바로보기가 매 세션 종료 시 동일한 작업을 반복하지 않도록 루틴화. 코니와의 싱크는 다음 세션 빠른 복귀를 위해 필수.
